# UX Polish Plan — 2026-04-03

Four tasks, ordered by user impact.

## Task 1: Unsave/unreact on matches page

**Problem:** Once a job is saved/skipped/applied, there's no way to undo it. The expanded action bar shows the current status as highlighted, but clicking it again does nothing — the API only accepts `saved|skipped|applied`, not a clear/reset.

**Fix:** Add a `clear` status option. Clicking an already-active status button clears it back to null.

### Backend

**File:** `shortlist/api/schemas.py` (modify)
- Change `JobStatusUpdate.status` from `Literal["applied", "skipped", "saved"]` to `Literal["applied", "skipped", "saved", "clear"]`

**File:** `shortlist/api/routes/jobs.py` (modify)
- In `update_job_status`: if `req.status == "clear"`, set `job.user_status = None`

### Frontend

**File:** `web/src/components/JobCard.tsx` (modify)
- Expanded action bar buttons: if the button's status matches `job.user_status`, send `"clear"` instead of the status. This toggles it off.
- Quick-action hover buttons on collapsed row: no change needed. They check `!job.user_status` and re-render via `loadData()` refetch after clear, so they reappear correctly.

### Tests

**File:** `tests/test_api_jobs.py` (create) — **non-async unit test** (pytest_asyncio not installed)
- Test that `JobStatusUpdate(status="clear")` validates
- Test that `JobStatusUpdate(status="invalid")` raises ValidationError

Note: full async API tests are in `tests/api/` but require `pytest_asyncio`. The schema validation test is sufficient for the backend logic. Frontend toggle is verified manually.

### Edge cases
- Rapid double-click: second click arrives before refetch → sends `clear` on stale `job.user_status` that's already null. Backend sets `user_status = None` (idempotent). Fine.
- Status badge on collapsed row disappears after clear (correct — badge checks `job.user_status`).
- Status counts on dashboard update after `loadData()` refetch (correct — counts come from server).

## Task 2: Profile page visual continuity

**Problem:** Steps 1-2 sit in one `divide-y` block, then AnalyzeButton breaks the flow as a standalone element, then steps 3-6 sit in a separate `divide-y` block. Feels like two different pages.

**Fix:** One `divide-y` container wrapping everything. Steps 3-6 are conditionally rendered inside it — Tailwind's `divide-y` only applies borders between adjacent rendered children, so this works cleanly.

### Frontend

**File:** `web/src/app/profile/page.tsx` (modify)

Structure:
```
<div className="divide-y divide-gray-200/60">
  <SectionCard step={1} ...>           {/* always */}
  <SectionCard step={2} ...>           {/* always */}
  <div className="py-6">               {/* AnalyzeButton section — always */}
    <AnalyzeButton ... />
  </div>
  {(hasProfile || generated) && <>
    {generated && <div>...banner...</div>}
    <SectionCard step={3} ...>
    <SectionCard step={4} ...>
    <SectionCard step={5} ...>
    <SectionCard step={6} ...>
  </>}
</div>
```

Key details:
- AnalyzeButton wrapper gets `py-6` to match SectionCard vertical rhythm. No `SectionCard` wrapper — it's a CTA, not a form section.
- Generated banner gets `py-4 px-1` inside the divide-y so it gets a top border like sibling sections.
- Remove the old centered "Search profile" divider label.
- Remove the "Re-analyze from resume" link below step 6 (the AnalyzeButton is always visible now).
- SaveBar stays outside the divide-y (fixed position, separate concern).

### Edge cases
- New user (no profile): only steps 1-2 + AnalyzeButton visible. Clean stopping point.
- After analysis: steps 3-6 appear with generated banner. The `divide-y` picks up the new children seamlessly.
- Fragment `<>` children inside `divide-y`: Tailwind's divide-y applies `border-top` to `* + *` CSS siblings. React fragments flatten, so SectionCards inside fragments become direct children. ✓

## Task 3: Save button disabled when clean

**Problem:** Save button is always clickable even when nothing changed.

**Fix:** Disable button when `!dirty`. Keep the bar visible (hiding it causes layout shift — page has `pb-28` to compensate for the fixed bar).

### Frontend

**File:** `web/src/components/SaveBar.tsx` (modify)
- Add `disabled={!dirty && !saving}` to save button
- Add `disabled:opacity-40 disabled:cursor-not-allowed disabled:hover:translate-y-0` to button classes
- Bar stays mounted always (when profile exists). Left side shows dirty indicator, toast, or error. Right side always has the button (enabled/disabled).

### Edge cases
- Toast after save: dirty=false, toast="Saved ✓". Bar visible, button disabled, toast visible. Correct.
- Error after save attempt: dirty may still be true (save failed). Button stays enabled. Correct.
- User makes change → save → more changes: dirty goes false then true. Button toggles. Correct.

## Task 4: History page redesign

**Problem:** Shows only status + timestamps. The `progress` dict has rich data but none is rendered. Near-zero user value.

**Fix:** Show what happened in each run.

### Progress dict shape (from `pipeline.py` + `worker.py`):

```typescript
// During run (flushed every 2s):
{ phase: string, detail: string, sources: { [name: string]: {
    status: "searching" | "filtering" | "fetching" | "scoring" | "done",
    collected?: number,
    filtered?: number,
    rejected?: number,
    scored?: number,
    matches?: number
  }}, matches: number }

// Completed:
{ phase: "done", detail: "Complete — X matches found",
  matches: number, jobs_collected: number }

// Failed: progress stays at last flushed state; Run.error has the message
// Cancelled: progress stays at last flushed state; no error
// Starting: { phase: "starting" }
```

### Frontend

**File:** `web/src/app/history/page.tsx` (rewrite)

**Layout per row** (follows DESIGN.md — `divide-y`, no cards):
```
grid grid-cols-[2rem_1fr_auto] gap-x-4 items-start py-5

[dot]  [main content]                              [duration]
       Today at 2:15 PM                             3m 42s
       42 collected → 18 scored → 4 matches
       {error if failed, amber text}
```

- **Status dot:** `w-2.5 h-2.5 rounded-full mt-1.5` — emerald-600 (completed), gray-400 (running/pending), red-500 (failed), gray-300 (cancelled)
- **Date:** `text-sm text-gray-900 font-medium` — "Today at 2:15 PM", "Yesterday at 9:00 AM", "Mar 28 at 4:30 PM"
- **Stats line:** `font-mono text-xs text-gray-400` — "42 collected → 18 scored → 4 matches" (from progress). Show what's available — if only `matches`, show that. If nothing, show "—".
- **Duration:** `font-mono text-xs text-gray-400` — computed from `started_at` → `finished_at`. Format: "3m 42s", "12s", "1h 5m". Show "—" if missing.
- **Error:** `text-sm text-amber-600 mt-1` — only if `run.error`. Truncated to 100 chars.
- **Running state:** stats line shows `progress.detail` ("Scoring jobs…") + pulsing dot

**Expandable per-source breakdown** (click row to expand):
```
  hn         12 collected → 8 scored → 3 matches
  linkedin   28 collected → 10 scored → 1 match
  nextplay   2 collected → 0 scored → 0 matches
```
- `font-mono text-xs text-gray-500`, left-padded under the main content column
- Only show if `progress.sources` exists and has data
- Source name in `text-gray-400`, numbers in `text-gray-500`

**Empty state:**
```
No runs yet.
Go to the dashboard and click Run to start your first search.
[Dashboard →]  (link)
```

### Null handling (every field is optional):
- `progress` empty/missing → show date + status dot only, stats line shows "—"
- `progress.sources` missing → no expand affordance (no chevron)
- `progress.matches` missing → omit from stats line
- `progress.jobs_collected` missing → omit "X collected" segment
- `started_at` null → no duration
- `finished_at` null + status=running → show "In progress" instead of duration
- `finished_at` null + status=failed → show "—" for duration

### No backend changes needed — all data already in `RunResponse.progress`.
